Guide

Task-based walkthroughs, full command reference, concrete do/don't examples, and every edge case OpenIdea's guard rails actually handle. For the 2-minute pitch, see the home page.

01How-to

Real scenarios, not just command syntax. Each assumes you already have the plugin installed.

Start a new project solo

  1. Run /openidea:init, describe the product in your own words. It writes openidea/BRIEF.md.
  2. Run /openidea:capture any time an idea comes to mind. Multiple ideas in one message are fine.
  3. Once a few ideas pile up, run /openidea:evaluate — it checks for overlap/conflict and recommends which are ready.
  4. Run /openidea:plan to group ready ideas into a milestone in ROADMAP.md.
  5. When openspec/ exists in the project, run /openidea:spec-draft <milestone> to generate the technical change.

No compile, export, or spec-audit needed — those only matter once someone outside the repo is involved.

Turn meeting notes into ideas

  1. Run /openidea:capture and paste in the transcript, MOM, or summary directly.
  2. It extracts candidate ideas as a one-liner list first — nothing is written yet.
  3. Review the list, say which ones to keep.
  4. Only approved ideas become files; the source document is archived to openidea/discovery/<session>.md and linked via each idea's source field.

Handle two ideas that overlap or conflict

  1. /openidea:evaluate compares Problem and Who uses it across all ideas — active, archived, and promoted, not just active ones.
  2. If it's genuine overlap (intersecting scope), it'll suggest merging or splitting.
  3. If it's a conflict (contradicting requirements from different sources), it won't guess a resolution — it flags it and asks you directly.

Run a client-facing revision cycle

  1. After /openidea:plan, run /openidea:compile <milestone> to generate a client-presentable narrative in proposals/v1.md.
  2. Iterate: each new compile writes a new version — v2.md, v3.md — old ones are never overwritten.
  3. Once the client signs off, tell Claude directly: "mark Milestone 1 approved from v3." No separate command for this.
  4. Run /openidea:spec-draft to generate the technical change from the approved scope.
  5. Anytime after, run /openidea:spec-audit <milestone> to check the OpenSpec output hasn't drifted from what was approved.

Hand off to a lead engineer without repo access

  1. Run /openidea:export <milestone>. It bundles relevant BRIEF.md context, the milestone's scope, the approved proposal (if any), and every related idea's full detail into one self-explanatory file.
  2. openidea/discovery/ is never included — export is safe to send externally even if your capture sessions contained sensitive client notes.
  3. Unlike compile, type: chore ideas are included — the engineer needs the technical debt items, even if a client never sees them.

02Reference

Every command: prerequisite, what it does, what it writes. Click to expand.

/openidea:initStructure a product vision into BRIEF.md▸
Prerequisite
None. If BRIEF.md already exists, a full rewrite requires your confirmation and archives the old version to discovery/brief-v{N}-superseded.md first — never a silent overwrite.
Process
Structures your free-form story into Vision, Target Users, Core Value Prop, Key Features, Non-goals, Constraints. Won't invent details you didn't mention — asks back on important gaps.
Output
openidea/BRIEF.md, schema_version: 1, project_status: active.
/openidea:captureCapture a new idea from a story or a document▸
Prerequisite
BRIEF.md must exist.
Process
Auto-detects free-form vs. document input. Checks new ideas against INDEX.md for near-duplicates before creating a file. Fills unclear fields with an explicit TBD and sets needs_clarification: true rather than guessing. One ambiguous item in a batch doesn't block the rest.
Output
New ideas/<slug>.md, status captured. Document mode also archives the source to discovery/. Regenerates INDEX.md.
/openidea:evaluateReview captured ideas, recommend status & category▸
Prerequisite
≥1 idea with status captured.
Process
Detects overlap and conflict across active, archived, and promoted ideas. Validates depends_on structurally — a direct cycle blocks, a broken reference just gets flagged. Recommends status/category; you approve before anything is written.
Output
Updated idea frontmatter. killed always gets a triage_note. Category is required before a status can move to ready. Final-status ideas move to archive/.
/openidea:planArrange ready ideas into ROADMAP.md milestones▸
Prerequisite
≥1 idea with status ready.
Process
Validates the milestone name yields a unique slug. Checks circular dependency at the milestone level, not just per-idea. Warns before renaming a milestone that's already been spec-drafted.
Output
openidea/ROADMAP.md. Moving an idea out of an approved milestone resets that milestone back to draft.
/openidea:spec-draftGenerate an OpenSpec change from a milestone▸
Prerequisite
Blocking: openspec/ must already exist (OpenIdea never creates it). The target milestone needs ≥1 idea ready.
Process
Reads BRIEF.md + ROADMAP.md + the approved proposal (if one exists) + each idea's full 5Q detail. Works fine with no proposals/ at all on the solo path. Won't overwrite an existing openspec/changes/<milestone>/ — offers to append or leaves it to a manual merge.
Output
openspec/changes/<milestone>/{proposal,design,tasks}.md. Related ideas move to ideas/promoted/, promoted_to filled in.
/openidea:compileCompile ready ideas into a client-facing proposal▸
Prerequisite
A ready idea already in ROADMAP.md.
Process
Groups ready ideas by category, orders per ROADMAP. Excludes type: chore — this document is for the client, not the engineer. Called 5+ times on the same milestone with nothing approved gets a light observation note (not a block).
Output
New openidea/proposals/vN.md. Old versions are never overwritten.
/openidea:exportPackage a milestone for handoff outside the repo▸
Prerequisite
None hard, but needs a milestone name.
Process
Bundles relevant BRIEF context, the milestone's ROADMAP scope, the approved proposal (if any), and full idea detail — type: chore included. Always excludes discovery/ and unrelated milestones. Optional language parameter translates the export only, never the source ideas/.
Output
openidea/exports/<milestone-slug>-vN.md, auto-versioned, self-explanatory for a reader with no AI.
/openidea:spec-auditRead-only drift check — OpenSpec vs. source ideas▸
Prerequisite
openspec/changes/<milestone> must exist.
Process
Checks the folder is complete (proposal/design/tasks) before comparing anything — incomplete gets reported as "in progress," not drift. Compares OpenSpec scope against the approved proposal + ideas for missing scope, added scope, and inconsistent acceptance criteria. Also checks whether the referenced OpenSpec change has since been archived/cancelled.
Output
A drift report. Never writes anything — not even on finding drift.

03Do / Don't

The patterns that matter most in practice.

Do

Let /openidea:evaluate change an idea's status. It updates history[] and enforces the category-before-ready rule.

Don't

Hand-edit an idea's status: field directly. You'll skip the history entry and any validation that would've caught a problem.

Do

Treat ideas/INDEX.md as a read-only view. Re-run any core command, or just ask, to regenerate it.

Don't

Edit ideas/INDEX.md by hand — it's fully rewritten from a filesystem scan on the next command, so manual edits are silently discarded.

Do

Let a vague capture fill unclear fields with TBD and set needs_clarification: true.

Don't

Ask Claude to fill in plausible-sounding target numbers, users, or scope it wasn't told. Guessed content is worse than an explicit gap.

Do

Run /openidea:spec-draft straight from ROADMAP.md + ideas/ on a solo project — compile/approval isn't required.

Don't

Force a compile → client-approval cycle when there's no client. It's overhead that exists only for the external-party path.

Do

Rename a milestone in ROADMAP.md before it's ever been spec-drafted — free to do, no side effects.

Don't

Rename a milestone after spec-draft already ran on it. The OpenSpec folder name won't follow — you'll get a mismatch between ROADMAP.md and openspec/changes/.

Do

Send a technical infeasibility finding through a normal /openidea:capture during a spec-audit.

Don't

Expect spec-audit to fix anything itself — it's read-only by design, always.

04Edge Cases

Blocking stops the command outright. Advisory flags a note but never blocks your workflow.

Running a command before BRIEF.md exists Blocking
Every command except /openidea:init and /openidea:export depends on project context existing. /openidea:capture stops immediately and points you to /openidea:init — it won't create an idea in a vacuum.
/openidea:spec-draft before openspec/ exists Blocking
OpenIdea deliberately never bootstraps openspec/ itself — that's a different tool's job. The command stops and tells you explicitly to run OpenSpec's own init first.
A milestone with zero ready ideas at spec-draft time Blocking
No empty proposals get generated. If nothing in the target milestone is ready, the command stops rather than producing a hollow proposal.md.
Two ideas with a direct circular depends_on Blocking
A → depends on B → depends on A is a structural error, not a judgment call. /openidea:evaluate stops and asks you to fix it before recommending any status. An indirect cycle (A→B→C→A) is caught the same way.
depends_on pointing at a slug that doesn't exist Advisory
Unlike a cycle, a broken reference doesn't block — it's flagged in the evaluate output and in INDEX.md's Anomalies section so you notice it, but doesn't stop you from moving other ideas forward.
An idea is needs_clarification: true when it's promoted Advisory
Nothing stops a vague idea from reaching ready or even promoted — the flag just travels with it. Once promoted, the open question typically resurfaces in design.md's open-questions section instead of INDEX.md, since the idea's no longer "active."
Moving an idea out of an already-approved milestone Advisory
/openidea:plan resets that milestone's status back to draft and clears approved_in automatically, with a notice explaining why — the client-approved scope no longer matches what's actually in the milestone.
Renaming a milestone that's already been spec-drafted Advisory
/openidea:plan warns before proceeding: the existing openspec/changes/<old-name>/ folder won't be renamed automatically, so the ROADMAP and OpenSpec folder name will diverge unless you handle that separately.
A merged_into target that's itself killed Advisory
/openidea:evaluate traverses the merge chain and surfaces this for review rather than leaving a dangling reference to a dead idea.
Running /openidea:spec-draft against a milestone folder that already has content Blocking (soft)
No silent overwrite of existing openspec/changes/<milestone>/ files. You're offered a choice — append as a note, or handle the merge manually — rather than the command picking for you.
An old openspec/changes/ folder was archived on the OpenSpec side Advisory
/openidea:spec-audit checks whether a promoted_to target still exists and is active. If OpenSpec has since archived or cancelled it, the idea's promoted status gets flagged for review — it may need to revert.
Adding a new idea in the middle of an /openidea:evaluate session Advisory
Fine to do — /openidea:evaluate isn't a locked mode. The new idea enters as captured but is deliberately not evaluated in that same pass, so it doesn't get judged half-formed.
A schema field from an old schema_version is missing Advisory
Every command is expected to be backward-compatible: fill missing fields with a safe default rather than failing outright on an older project.

05Why it's built this way

The design decisions that shape everything above.

Why markdown files instead of a database or app

State that lives in git is diffable, mergeable in the ordinary git sense for most files, readable without any tooling, and survives the plugin itself being uninstalled. The one exception is ideas/INDEX.md — it's regenerated wholesale from a scan, so a git conflict on it is resolved by regenerating, not merging.

Why Claude never changes status without asking

Claude structures and recommends; every status transition, milestone approval, and full-rewrite is a human decision. This is why /openidea:evaluate presents recommendations and waits, rather than writing them straight to disk.

Why OpenIdea won't bootstrap openspec/ itself

OpenSpec is a separate, independently-versioned tool. OpenIdea reads from it (to check whether a folder exists) and writes to it only through the single, explicit /openidea:spec-draft path — never as a side effect of any other command.

Why discovery/ is excluded from every handoff

Raw meeting notes and transcripts can contain sensitive client information that was never meant to be redistributed. compile, export, and proposals/ are built to never include it, without exception — not even partially.

Why nothing is ever deleted

killed, merged, and split ideas move to archive/ with a reason, not the trash. A decision that turns out to be wrong can be revived with a new history[] entry — the record of the earlier call stays intact instead of vanishing.
